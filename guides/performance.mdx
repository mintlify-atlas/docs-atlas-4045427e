## Complexity

| Operation | Complexity | Notes |
|---|---|---|
| `alloc` (fast path) | O(1) | Single bounds check + bump pointer advance |
| `alloc` (new block) | O(1) amortized | Block list appended, not rebuilt |
| `checkpoint` | O(1) | Copies 3 integers |
| `rewind` | O(k) where k ≈ 0–1 | Typically one block touched |
| `reset` | O(p) where p = peak blocks since last reset | Only used blocks zeroed; untouched blocks skipped |
| `stats` | O(1) | Reads pre-maintained counters |

## Why fastarena excels

- **5–8x faster slice allocation** than bumpalo — batch write in a tight loop, no per-element iterator overhead
- **2–3x faster ArenaVec** than bumpalo/typed-arena for bulk collection building
- **Tied on single-item alloc** — on par with bumpalo for individual `alloc` calls
- **14x faster than Box** for bulk alloc + reclaim cycles
- **35x faster reset** than bumpalo for single-block arenas, thanks to O(peak) block reuse with no OS calls
- **Zero dependencies** — no external crates required

<Info>
  The `alloc_str` benchmark is the one case where bumpalo edges ahead (190 ns vs 202 ns). For workloads dominated by string interning, the difference is negligible in practice.
</Info>

## Tuning tips

**Choose initial capacity wisely.**
Pass a size to `Arena::with_capacity` that comfortably fits a typical workload. This avoids mid-run block chaining and keeps all allocations in one contiguous region.

```rust
use fastarena::Arena;

// Size the arena to your workload's expected high-water mark
let mut arena = Arena::with_capacity(1024 * 1024); // 1 MiB
```

**Single-block arenas pay O(1) reset.**
If your workload fits in the initial block, `reset()` zeroes exactly one offset — effectively free. The `high_water_mark` tracker ensures only used blocks are touched.

```rust
use fastarena::Arena;

let mut arena = Arena::with_capacity(256 * 1024); // fits all requests

for req in requests {
    handle(&mut arena, req);
    arena.reset(); // O(1) when single-block
}
```

**Use `alloc_cache_aligned` for SIMD workloads.**
Allocates with 64-byte cache-line alignment — required for AVX-512 and beneficial for any SIMD load/store.

```rust
use fastarena::Arena;

let mut arena = Arena::new();

// 64-byte aligned — safe for SIMD operations
let buf = arena.alloc_cache_aligned(512);
assert_eq!(buf.as_ptr() as usize % 64, 0);
```

**Use `alloc_slice_copy` for `Copy` types.**
A single `memcpy` instead of an element-wise iterator. This is the fastest path for copying numeric slices.

```rust
use fastarena::Arena;

let mut arena = Arena::new();

let src: &[f64] = &[1.0, 2.0, 3.0, 4.0];
let dst: &mut [f64] = arena.alloc_slice_copy(src); // single memcpy
```

**Use thread-local arenas for zero-contention request handling.**
`Arena` is `!Send + !Sync`. Give each thread its own arena via `thread_local!` and you get allocation with no locks, no atomics, and no cache-line bouncing between cores.

```rust
use fastarena::Arena;
use std::cell::RefCell;

thread_local! {
    static ARENA: RefCell<Arena> = RefCell::new(Arena::with_capacity(64 * 1024));
}

fn handle_request(data: &[u8]) {
    ARENA.with(|cell| {
        let mut arena = cell.borrow_mut();
        // allocate freely — no contention with other threads
        let buf = arena.alloc_slice_copy(data);
        process(buf);
        arena.reset();
    });
}
```

<Tip>
  Combine thread-local arenas with per-transaction budgets (`txn.set_limit(N)`) to guard against runaway allocations on a per-request basis without any shared-state overhead.
</Tip>
